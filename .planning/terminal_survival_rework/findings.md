# 现状发现与设计依据

## 2026-07-21 频段冗余的玩法结论
- 玩家实机试玩后确认：四页终端已完全承担信息架构，旧固定频段只剩下开局校准门槛和视觉装饰，不再支撑长期决策。
- 现有主页把“听清三个有声音的频率”显式做成 `calibrate 0/3`，实际交互只是在固定位置稳定 20 Tick，是一次清单操作。
- 校准完成后默认打开自动调频，服务端每 80 Tick 在已知电台间轮换；这使手动频段在教学后立即失去作用。
- 记录页已通过 `allSignalEntries()` 展示全部事件，不依赖当前频段；天气、矿物和公共记录也已被专用工具或记录页取代。
- “信号”工具直接显示 `SignalBand.UNKNOWN` 内容，与全局滑块调到四频段形成重复入口。
- 唯一仍然有明确玩法价值的部分，是玩家进入原版建筑范围后，根据当地目标频率锁定 20 Tick 并触发短异常/碎片。这一部分应保留为“近场接收器”，不再承担全局内容导航。
- 新的概念边界：“第四频段”不是第四个可选电台，而是逐渐污染终端工具、记录和波形的异常信号层。

### 调用与兼容盘点
- `TerminalRuntimeService` 每 Tick 对打开的终端先执行自动轮台，再执行频段/碎片锁定；全局 `TUNE` 现在只验证 0..100，没有校验玩家是否打开“信号”工具。
- 附近碎片目标会把主页 objective 全局替换为 `nearby_signal`，并借用 objective `target` 传递隐藏频点；这会让附近信号覆盖正常生存任务。新实现应把接收器状态移入工具快照，主页 objective 不再被劫持。
- `TerminalRuntimeService.acknowledgeLockedBand` 同时承担“公开台校准/已读”和“附近碎片完成”。实施时应拆成单一的近场接收器流程，并仅在 `selectedTool == SIGNAL` 时计时。
- 当前四组碎片目标使用固定数值 `{28,55,73,95}`；这仍是四个可记忆的隐形电台。新方案应从候选建筑的稳定 key/位置派生局部峰值，不再按碎片编号固定。
- `StoryProgressService` 的绑定条件是“三频校准 + HOME/IRON 任一”，而签名异象必须在 IRON 后才发生。因此可安全将绑定改为 `HOME || IRON`，同时保留 IRON 后才揭示异常信号层的节奏。
- `maintenance_handoff` 文件目前只由三频校准解锁，需改挂 `BOUND` 或首个生存节点；否则移除校准后此文件永不出现。
- 新建终端会先写入环境、资源和发放三条分频初始记录；新流程应只写一条终端发放/自检记录，旧存档的三条历史记录保持可读。
- `SignalBand` 仍广泛用作旧日志的稳定 wire ID、每组上限和测试分类。第一轮不应删除 `band` 存储；应只删除它与调频、玩家筛选和校准的关系。
- 四页 UI 没有告诉服务端当前是 HOME/TOOLS/RECORDS，它们都映射到 SIGNAL wire mode。近场锁定可直接借用已有 `SELECT_TOOL` 的服务端 `selectedTool`；记录已读则需新增一个明确受校验的 `MARK_RECORDS_READ` 动作。
- `TerminalToolSnapshotPayload` v1 只有附近信号数量，没有接收器目标、当前强度或锁定进度。应升为 v2，新增 `receiverAvailable`、`receiverTarget`、`receiverStrength`、`receiverLockTicks`；终端快照 v5 无需变更。
- 终端快照 v5 中的 `publicStationMask`、`tuning`、`reminderBand` 可先保留字段形状，分别固定发送 0、仅表示局部接收器数值、-1；客户端不再把它们解释为固定电台。
- 工具快照 v1 中的 `automaticTuning` 不应复用为其他含义。升到 v2 时可移除该字段；存档 `AUTO_TUNING` 则继续容错读取但停止新写。
- 终端右侧现有滑块、自动/手动按钮和 LCD 有独立安全坐标。可在不改动四页主布局的前提下，平时不画滑块/按钮，只在“工具 > 信号”且附近有目标时复用滑块坐标画近场接收器。
- `TerminalSignalLog` 目前只能按 band 已读，且每 band 保留 32 条。第一轮保留分组上限以兼容旧数据，新增 `markAllRead`；读取历史时不改写旧 `band` 值。
- 当前“信号”工具仅读取 `SignalBand.UNKNOWN`，但四组碎片候选被分散写入 WEATHER/MINING/PUBLIC/UNKNOWN，因此工具实际只能完整显示第四组。新工具必须按 `fragment_*`/剧情事件类型聚合全部 band，不再依赖 band 过滤。
- `FragmentInvestigationService` 的候选键已含 fragment、group 和位置，可不增加存档字段，直接用这些稳定数据派生 12..88 范围内的局部峰值。服务端仍在完成时重新核对真实建筑范围和误差≤2。
- `TerminalSignalLog.entries()` 按 sequence 倒序返回，但主页 `latestSignalEvent()` 使用 `getLast()`，实际取到最旧一条。频段退场时应顺手改为 `getFirst()`，否则“最近发生”仍会误导玩家。
- `BAND_STAGE` 不只是旧 UI 频段：它还门控观察者前奏、四碎片候选分配、候选记录显示、异象强度和结局停止。第一轮不删此字段，只把其玩家含义改为“异常信号层阶段”；内部 ID 可等未来 schema 重构再改名。
- 持久化不需升级 schema v5：本次不增加存档字段，旧的 `CALIBRATED_BANDS_MASK`/`AUTO_TUNING` 保持容错读取和默认值，但不再参与任务、绑定或交互。只有工具网络快照从 v1 升到 v2。
- 现有测试中至少有 3 组纯单元契约、6 个服务端 GameTest 方法和 mainline 客户端流程直接假定固定电台/校准/自动调频。这些必须改为近场接收器、自然绑定和全局记录已读契约，不能仅删断言。

## 2026-07-21 实施后的确认

- 生产实现与最初盘点一致：绑定只读 HOME/IRON；签名场景决定异常信号层揭示；附近信号不再覆盖主页 objective。
- 工具快照 v2 已成为接收器状态的唯一 S2C 来源。客户端虽然收到 `receiverTarget` 以计算本地反馈，但不直接显示目标；完成仍由服务端误差、连续 Tick 和结构范围三重核对。
- 旧 `SignalBand` wire ID 必须继续保留，因为历史日志仍按该值解码；删除的是玩家可见分类、锁台与校准语义，不是存档字节含义。
- 客户端回归的首轮锁定音失败由夹具残留固定值 55 导致。将其改为候选确定性峰值后主线完整通过，证明测试也不能再假设碎片编号对应固定频率。
- 最终证据为 JUnit 91/91、服务端 GameTest 38/38、客户端 mainline 1 分 53 秒成功；当前无未解决实施阻挡。

## 2026-07-21 当前 MOD 流程源码复核
- 当前生产入口仍为 Fabric 公共端 `TheFourthFrequency` 与客户端 `TheFourthFrequencyClient`；公共端集中注册世界、终端、异象、校正体、设施和祭坛服务，客户端集中注册一次性告示、Alpha 降级、终端 UI、异象表现与 Meta 恢复。
- 客户端第一次到达标题页先显示一次性安全告示，确认状态写入客户端本地配置；第一次进界触发一次加载崩坏并写入另一份客户端本地状态，后续启动直接预选 Programmer Art、Golden Days Base、Golden Days Alpha 并恢复 Alpha 标题/图标，不重播崩坏。
- 新世界服务端在原出生点分 Tick 建造唯一零号站并改写世界出生点；每个玩家首次加入只发一台带世界 ID、终端 ID、所有者和复制代次的个人终端，并传送到零号站。重连、死亡、重复副本与丢失由终端生命周期服务回收或去重。
- 手持本人终端右键后，客户端发送打开请求；服务端重新校验手、物品、所有者、世界 ID、终端 ID 与复制代次，然后发送终端快照、工具快照和导航快照。当前 UI 是主页、工具、记录、文件四页；底层仍复用 SIGNAL/FILES 两种 wire mode。
- 当前引导任务顺序为：三个公开频段校准 → 住处 → 铁 → 下界准备 → 进入下界 → 返回主世界 → 制作末影之眼 → 至少两次真实投眼 → 靠近记录的要塞 → 找到真实传送门房 → 填满门框启动祭坛 → 最终战 → 完成。
- 三个公开频段需在对应频点稳定 20 Tick 才计入校准；全部完成后默认开启自动调频。终端任务是引导而非祭坛硬门槛：祭坛入口实际只校验真实要塞、完整门框条件、有效个人终端和中心可放置装置，未逐项校验前置任务或绑定状态。
- 绑定实际条件为“三频校准 + 已有住处或铁任一项”；主页任务仍会按固定顺序继续要求缺少的住处/铁。铁里程碑会生成一次异常痕迹、暂时停用工具并激活校正体；第四频段在该签名场景启动后揭示，不等待校正体完成拆除。
- 校正体只以异常痕迹、旧信号封壳和祭坛装置为工作目标；平时不捕食玩家。玩家攻击、持续挡路 40 Tick，或在工具停用期强行操作时，才会触发约 20 秒敌对。异常痕迹被拆除、或校正体带着该目标死亡后，工具恢复并追加解释记录。
- 绑定后启用单玩家单实例异象调度；强度上限由住处/铁、下界往返、文件/裂缝、要塞和终局逐级提升到 1–5。异象在睡眠、终端打开、空段事件或已有异象时不启动；结束清理后才写入终端记录。
- 第四频段揭示后，世界开始为四个共享碎片分配原版建筑候选。玩家到候选建筑附近，在对应频点误差 2 以内保持 20 Tick 即取得碎片；碎片和完成状态全服共享。四片齐全后自动解锁前人文件并建造两界裂缝支线。
- 结局关键值不是“文件已解锁”，而是玩家在文件页实际打开已解锁的 `encrypted_witness_file`，服务端才写入 `TRUTH_READ`。祭坛启动时把启动者的该值冻结为全局 `truth_read_at_start`，启动后再读文件不会改变本次结局路线。
- 真实要塞门框填满后不会进入末地；所有末地传送门方块的实体传送都被 Mixin 全局取消。门框中心改为四个祭坛装置并生成最终实体，实体只攻击同维度、祭坛 48 格内玩家；离开范围可暂停，不追基地、不夺终端、不吸收或放置地形。
- 最终实体存活时每 600 Tick 自动移除一个祭坛装置，校正体也把装置视作最终工作目标；玩家也可直接拆除装置。击败最终实体时：未在启动前读真相为 `undiscovered_truth`；已读且剩余装置少于 2 个为 `prevention_failed`；已读且至少 2 个为 `prevention_succeeded`。
- 三种结局都会停止活动异象并写入所有终端。只有成功结局把世界衰败阶段归零并追加 `weak_resonance`；另外两种结局仍会按已完成生存里程碑保留较高世界衰败表现。
- 持久化实际为 schema v5：世界级 SavedData 保存零号站、发放账本、共享碎片/设施/校正/祭坛/结局；每玩家终端记录保存里程碑、工具、文件、真相和日志。终端快照 v5、导航 v4、工具快照 v1、异象 v3、Debug v2。

### 当前遗留与不一致
- `README.md` 仍描述 SIGNAL/FILES 双页、无常驻任务、schema v4 和旧测试数字，已落后于当前四页 UI、主页任务与 schema v5。
- `BodyConstructionService`、`BODY_PROGRESS`、`BODY_STAGE`、production pacing 等名称/字段仍存在，但生产主线已由生存里程碑驱动；前者当前主要承担下界裂缝与连续性支线，旧字段主要用于存档/协议兼容。
- 旧五设施密码路线仍保留实现，但 `FacilityService.ensureAllocated` 没有生产调用；新世界实际走四碎片自动解锁路线，旧设施更像存档兼容残留。
- `FinalConfrontationService.useTerminationSpike` 没有任何生产调用点，终止钉也是普通 `Item`；当前玩家可合成但不能通过正常实体交互触发其打断/削弱逻辑。
- 祭坛和结局是世界共享状态，但 `PORTAL_ROOM_FOUND` 是每玩家字段。祭坛启动/结局写入没有为所有终端补齐该字段，因此未到过传送门房的其他玩家可能在全局结局后仍看到“寻找房间”的早期任务。
- 祭坛入口不硬校验校准、绑定和里程碑；速通玩家可凭有效终端与真实门框直接启动并完成终局，但其个人主页仍可能停留在更早任务。

## 已确认需求
- 终端保留明确的当前任务和进度提示。
- 终端必须真正方便生存，而不是只有剧情日志。
- 工具页类似手机 APP 桌面，点击后在同一页进入详细信息。
- 文件页在主显示区内部采用 30/70 分栏；左侧逐行显示短标题，右侧阅读正文。
- 每次刚进入文件页时不选中任何文件，右侧显示“选取文件来查看”。
- 玩家可见文字不用抽象系统词，改为看到就懂的日常表达。
- 调频默认自动，手动调频只在少量重要场景使用。
- 玩家以击败末影龙为生存目标，但最终在要塞被第四频段截断，不真正进入末地。

## 初步源码发现
- 当前终端为 512x256 基准布局，主显示区与右侧示波器、罗盘、调频滑块、LCD 已明确分区。
- 当前客户端页面只有 SIGNAL 和 FILES 两种服务端模式，文件内部还有目录、详情、碎片和密码视图。
- 当前文件目录为三列两行卡片，已经具备选择、鼠标滚轮、方向键、Enter 和详情滚动逻辑。
- 当前快照协议为 v5，已包含任务 ID/进度、活动异象、文件、信号日志和部分导航数据。
- 当前导航包 v4 只支持铁、红石、钻石和结构碎片四种目标；住处、上次传送门、天气和要塞估算尚未接入。
- 当前资源寻找不是玩家主动选择：服务端按背包缺口自动选择铁、红石或钻石，在半径 48 内真实分 Tick 扫描。
- 当前剧情绑定后会停止资源扫描，并且导航服务会隐藏除结构碎片外的目标；新工具页若要贯穿生存主线，必须解除这两个旧限制或增加新的明确规则。
- 终端打开时服务端保存的仍是 SIGNAL/FILES 两种 mode；主页、工具和记录可作为客户端子页面复用 SIGNAL，文件页再切换服务端 FILES，从而避免仅为页面导航升级 v5 快照协议。
- 终端每 20 Tick 同步快照、每 4 Tick 同步罗盘数据，足以支持详情页中的动态方向和距离。
- 当前身体生成进度仍参与快照、异象强度和最终流程，不能只删除 UI 文案。
- 当前任务服务已经按“调频、行为证明、前期异象、设施、文件、裂缝、下界往返、身体生成、Boss”返回单一任务和进度；新主页可复用它的传输形状，但任务顺序与名称必须整体换成原版生存节点。
- 当前文件界面只使用一个 `scrollRow`，目录和正文共享滚动状态；30/70 分栏需要拆成左侧列表滚动与右侧正文滚动两个独立值。
- 当前文件点击后会离开目录进入全屏详情；新设计需要改为保持左侧列表并在右侧直接更新正文。
- 加密见证文件目前是特殊容器，会进入四份碎片页，完整后还可能进入密码键盘；新分栏必须把碎片、密码和完整正文都放进右侧，不再替换整个文件页。
- 当前 `resetLogView()` 在进入 FILES 时已经把选择重置为 -1，可直接延续“刚进入文件页右侧为空”的规则。
- 现有静态契约明确禁止持久任务清单，并检查旧三列文件网格；用户现已明确允许单一当前任务，因此这些测试必须有意识地改写，不能被旧断言误判为回归。
- 身体生成不是孤立模块：它控制最终 Boss 自动唤醒、异象最高强度、世界变化、玩家行为采样、终端外观、文件发现、私有异象载荷和调试状态。移除路线需要先引入新的生存节点状态，再逐项替换依赖，旧字段只保留迁移读取。
- 当前存档只保存最近一次传送的起点和终点维度名称，没有保存传送门方块位置；“上个传送门”工具需要新增真实位置和所在维度字段。
- 当前天气记录只保存最近天气种类，不提供预测时间；第一版天气工具可可靠显示当前天气和昼夜，预测功能必须单独实现，不能伪造。
- 当前没有住处位置、末影之眼轨迹或要塞估算字段；这三个工具都需要新的服务端采集与持久化。
- 当前校正体只会在后期校正目标已激活且附近有玩家时生成；没有目标时会直接追逐任何携带终端的玩家并造成伤害，因此既难在前期看见，也不符合“先修复、玩家妨碍后才敌对”的新机制。
- 当前校正体只识别造身器官和终端设施两类方块目标。删除造身环节后，需要把目标改为可恢复的异常痕迹、信号封壳或最终封锁装置。
- 当前文件目录共有 12 个稳定 ID；可以保留 ID 与发现状态，只改短标题和正文含义。其中 `body_mapping_warning` 需迁移为“它在找入口”等新叙事，不能删除 ID。
- 当前最终 Boss 会在任一绑定玩家身体进度达到 1000 后自动在其附近出现；未知道真相会立即锁定未发现结局，失败结局依赖终端被夺走，成功依赖终止钉。要塞祭坛、固定装置存活数量和最终战入口均需新状态。

## 兼容边界

## 阶段 6 实施核对
- 生产入口只接受真实要塞中的完整、朝向正确的 12 个原版末地门框；最后一只末影之眼或手持有效终端的明确交互才会启动祭坛，普通玩家附近和旧 `BODY_PROGRESS=1000` 都不会生成最终实体。
- `EndPortalBlockMixin` 让末地传送门方块失去传送作用；祭坛服务也会清掉门框中央可能生成的传送门方块，因此玩家不会进入末地。
- 祭坛状态保存在世界叙事数据中，包含维度、中心、启动玩家、启动时是否读过真相、四个装置位置、剩余数量、最终实体 UUID、阶段和时间；区块未加载时不会强行加载或继续破坏装置。
- 四个祭坛装置是独立注册的真实方块，玩家、最终实体和校正体都能改变其实际存活状态；校正体拆除装置时只移除装置本身，不在要塞周围留下伤痕或支架。
- 最终实体只选择同维度且距祭坛 48 格内的存活玩家，不跨维度追逐，不夺取终端或物品，不吸收地形，不放置阻路方块；离开祭坛可安全暂停，不会追到基地。
- 三结局仅使用祭坛启动时的 `truth_read` 与击败实体时真实剩余装置数：未读真相为 `undiscovered_truth`；已读且少于 2 个为 `prevention_failed`；已读且至少 2 个为 `prevention_succeeded`。
- 三条路线都以击败最终实体收束，并把任务切到 `complete`；成功结局只追加微弱共鸣能力，不抹掉玩家此前获得的终端工具能力。
- 终端任务顺序已经接入 `find_portal_room`、`open_altar`、`defeat_entity`/`protect_anchors` 和 `complete`；祭坛发现、启动、装置数量、实体出现/击败和结局都有服务端记录。
- `truth_read` 只能由服务端确认稳定的 `encrypted_witness_file` 已发现且解锁后写入；客户端不能直接声称已读真相。
- 调试面板只提供祭坛 0..3 和结局 0..3 的固定枚举；旧任意 Boss 生成/移除入口不再暴露给面板。稳定文件 ID、结局 ID、旧结局字符串和历史日志读取保持不变。
- 终端快照仍为 v5，导航仍为 v4，工具快照仍为 v1；阶段 6 没有新增半接入网络协议。
- 客户端截图确认祭坛任务文字、四个真实装置、最终实体/Boss 条和完成任务均正确显示；客户端场景使用明确测试夹具，真实要塞校验仍由生产服务端入口负责。
