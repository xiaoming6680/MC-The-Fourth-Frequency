## 关键问题
1. 现在实际会生成哪些配置文件，分别属于什么加载范围？
2. 统一配置是否需要同时覆盖客户端、通用和服务端选项？
3. 项目对 JUnit、GameTest 和客户端回归的执行是否有特别限制？

## 已做决策
| 决策 | 理由 |
|---|---|
| 不保留旧配置或旧世界迁移层 | 用户明确 MOD 尚未发布，可直接使用干净新结构 |
| 使用 .planning/consolidate_mod_config/ 独立记录本轮任务 | 避免混入已完成的终端重构计划 |
| 唯一文件保留为 config/thefourthfrequency.json | 名称已直观且无需引入额外目录 |
| 根配置按 meta、pacing、limits、clientState 分组 | 兼顾单文件目标与可读性 |
| ConfigManager 独占全部 JSON 与原子写入 | 消除客户端状态类中的重复持久化实现 |

## 遇到的错误
| 错误 | 尝试次数 | 解决方案 |
|---|---:|---|
| 系统 python 命中 Windows Store 占位符，无法运行会话恢复脚本 | 1 | 加载 Codex 工作区依赖后改用内置 Python，恢复检查成功 |
| 两次补丁包装分别触发 JavaScript Unicode 转义与模板字面量解析错误 | 2 | 改用相对路径、原始字符串，并移除内容中的反引号 |
| 首次并行盘点因 AGENTS.md 无匹配返回退出码 1 | 1 | 后续命令显式容忍 ripgrep 的无匹配状态 |
| 测试与文档合并补丁因 Markdown 反引号触发模板解析错误 | 1 | 拆分补丁并使用占位符写入文档反引号 |
| 完整 unitTest 的 101 项中有 1 项中文术语检查失败 | 1 | 已定位为未改动的 zh_cn.json 中多处“异常”旧用词；配置专项 4/4 通过，本任务不扩张修改文案 |
| 首次 JAR 条目审计找不到 jar 命令 | 1 | 改用 PowerShell 的 ZipFile API 读取 JAR，确认旧类条目为 0 |
| 完成检查的 PowerShell 脚本先被执行策略阻止，绕过后又因技能文件编码损坏无法解析 | 2 | 改用同技能的 check-complete.sh，并显式传入活动计划；确认 5/5 阶段完成 |
