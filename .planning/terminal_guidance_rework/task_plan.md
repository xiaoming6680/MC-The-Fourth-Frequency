# 任务计划：终端主线引导与智能工具重构

## 目标
保留终端调频滑块及其音效，但让前期拖动不产生玩法效果；以“采集与验证数据”的直白文案重构通往末地异常 BOSS 的不可跳过主线；让首页按玩家状态智能推荐工具、工具详情替换首页中部卡片并可正确返回首页；把“信号”改为渐进披露的结构导航；修正档案页命名、进度条动画和六个终端工具图标，使 3 区块视距下的引导足够可靠。

## 已做决策
| 决策 | 理由 |
|---|---|
| 使用独立 `.planning/terminal_guidance_rework/` 作用域 | 保留已完成的终端生存重制和视距/雾效记录 |
| 任务节点必须由真实游戏行为驱动 | 避免纯计时、按钮或可跳过提示冒充主线进度 |
| 导航与工具能力由服务端判定 | 防止客户端伪造结构位置、解锁状态或任务完成 |
| 调频滑块前期只保留音效与机械反馈 | 尊重终端视觉身份，同时不让旧调频系统干扰新主线 |
| 主线固定为原木 7→铁→进入下界→烈焰棒 6→返回→末影之眼→两次投眼→要塞→进入末地→击败末影龙 | 每个节点都有真实生存行为或服务端事件，文案保持数据采集感且无需理解抽象系统词 |
| 停止新祭坛入口并移除末地传送拦截，旧祭坛实现只保留恢复/调试能力 | 任务目标若是末影龙，原版末地必须真实可达；同时避免粗暴删除旧存档恢复代码 |
| 首页保持 HOME 页面；默认一个首要建议，停滞或紧急状态才追加第二项；点击后用跨两栏实时卡替换两张卡 | 避免信息一次铺满，同时满足智能推荐、简约实时更新和从首页返回首页 |
| 工具网格始终显示六个工具，未解锁项可查看条件但不能发起服务端动作 | 玩家能预期后续能力，又不会开局直接使用导航或钻石扫描 |
| 住处/天气开局可用；原木完成后解锁矿物/导航；铁后若卡住才披露钻石，红石更晚；进入下界后解锁传送门；末影之眼后解锁要塞 | 能力解锁与内部目标披露分离，避免开局或刚解锁就展示全部内容 |
| 导航按任务只给一个主目标，在线停滞约 2/5 分钟后依次补充备选/支线；不稳定来源还要求至少进入过下界 | 在 3 区块视距下提供必要方向，同时避免过早暴露异常支线或分散注意力 |
| 任务进度变化立即清零停滞计时，离线不累计，当前已选导航继续保留 | “卡住”来自真实在线停滞，不会因离线或提示收束导致目标丢失 |
| 末地传送门作为祭坛入口；进入末地后以“误读实体”替换末影龙作为最终 BOSS | 保留不可跳过的原版生存路线，同时让终局与模组叙事一致 |
| 面向玩家将塔顶装置统一称为“稳定锚” | 名称直观表达其抑制作用，并避免直接说出“末影水晶” |
| 倒计时归零只锁定失败结局，不移除 BOSS 或强制退出战斗 | 满足“倒计时不是真正倒计时”；玩家仍可继续战斗，但无法把失败翻回成功 |
| 稳定锚越少，移动/攻击/减伤/适应能力按固定档位单调增强；零锚仅保留极低可击败性 | 保证玩家不能通过清空装置把战斗变简单，同时避免绝对无敌造成永久软锁 |

## 遇到的错误
| 错误 | 尝试次数 | 解决方案 |
|---|---:|---|
| 初次并行目录盘点因一个无匹配搜索返回非零而整组结果未展示 | 1 | 改用容错并行读取并为无匹配显式返回成功状态 |
| PowerShell 文件统计在 `foreach` 块后直接使用管道导致解析失败 | 1 | 先把循环结果赋给 `$rows`，再单独格式化输出 |
| 首次 `javap` 查询只找到 JDK，通配目录未解析到 Minecraft JAR | 1 | 从已确认的 Loom `minecraft-common` 缓存根目录精确选取 named JAR |
| 一次 `TerminalToolService` 大块补丁因上下文变化无法匹配 | 1 | 拆成导入、快照、资源门槛和导航分支的小补丁逐项应用 |
| 一次运行时与协议合并补丁因精确上下文不一致失败 | 1 | 分文件读取现状后分别应用，避免跨文件大上下文匹配 |
| 一次同时修改三份规划文件的补丁因表格上下文不完全一致而整体失败 | 1 | 先用 UTF-8 明文读取确认实际内容，再按文件拆分应用 |
| 渐进披露首次编译引用了终端界面不存在的 `DARK_GREEN` 常量 | 1 | 改用现有 `DIM` 调色板常量；随后四源集编译成功 |
| 全量 JSON 校验先后遇到 `ConvertFrom-Json` 拒绝合法空属性名、Windows PowerShell 无 `System.Text.Json`、系统 Python 别名无解释器 | 3 | 使用 Codex 工作区内置 Python 标准库解析模组自身 47 份 JSON，全部通过 |
| 规划完成检查脚本先受执行策略阻止，改为脚本块后又因 Windows PowerShell 默认 ANSI 读取 UTF-8 计划而统计为 0/0 | 3 | 以 UTF-8 读取脚本并为 `Get-Content` 设置 UTF-8 默认值，最终确认 6/6 阶段完成 |
